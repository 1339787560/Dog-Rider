```yaml
Skill_Name: Code-Planner
Purpose: 编程任务敲定：需求拆解 + 实现方案 + 测试方案
Execution_Trigger: 用户提出代码实现/开发/修改需求，或 Agent 需要系统性规划编程任务时激活
```

## 概要

本 Skill 用于编程任务的前期规划阶段，确保需求理解完整、方案清晰、测试覆盖到位。

1. 需求拆解：逐条列出用户需求点，识别盲区
2. 实现方案：用 TodoWrite 输出可执行的任务清单
3. BDD 场景定义：从用户视角定义行为预期，确保与需求对齐
4. 测试方案：明确单元测试、集成测试的范围和方法

---

## 详细规范

### 详细规范 1：需求拆解

- 必须逐条列出用户明确提出的需求点（编号 1、2、3...）
- 每条需求点不超过 30 字，只陈述事实，不加主观判断
- 必须识别需求盲区：对实现方式、约束条件、边界情况不明确的地方，单独列出"待确认问题"
- 待确认问题语调必须柔和：用"请问..."、"能否说明..."，禁止用"必须提供"、"缺少"等强硬措辞
- 待确认问题超过 3 个时，分优先级标注：[高]（不确认无法继续）、[中]（可先假设，后续确认）

#### 示例

```
需求点：
1. 实现用户登录 API
2. 密码用 bcrypt 加密
3. 返回 JWT token
4. 登录失败次数超限后锁定账户

待确认问题：
[高] 请问密码长度和复杂度要求是怎样的？
[高] 请问登录失败多少次后锁定？锁定时长是多久？
[中] JWT token 的有效期和签发者需要配置吗？
```

### 详细规范 2：实现方案

- 必须使用 TodoWrite 格式输出任务清单
- 每个任务必须包含：Todo 内容 + 预估修改范围（文件名/模块）
- 任务数量控制在 2-8 项，超过时合并同类项
- 必须标注任务之间的依赖关系（用箭头 A → B：A 完成后才能做 B）
- 涉及数据库改动、API 合约变更、配置项新增等高风险项，必须前置标注 `[高风险]`

#### Todo 格式

```
## 实现方案

[ ] 1. 定义用户数据模型 + 数据库迁移脚本
    → 涉及文件：src/models/user.py, migrations/xxxx_add_user.py
    → [高风险] 数据库结构变更

[ ] 2. 实现密码加密工具函数
    → 涉及文件：src/utils/crypto.py, tests/test_crypto.py

[ ] 3. 实现登录 API 接口 + JWT 签发
    → 依赖：1、2
    → 涉及文件：src/api/auth.py
```

### 详细规范 3：BDD 场景定义

- 必须从用户视角定义行为场景，确保实现与用户预期一致
- 每个场景必须包含：Given（前置条件）→ When（触发动作）→ Then（预期结果）
- 必须覆盖：正常路径、异常路径、边界路径
- 场景描述必须使用业务语言，避免技术术语（用户能看懂）
- 场景数量：核心功能 ≥ 3 个，复杂功能 ≥ 5 个
- 场景编号用 SC01、SC02、SC03...

#### BDD 场景示例

```
## 3. BDD 场景定义

### SC01 - 登录成功
**Given** 用户名 "alice" 已注册，密码为 "secret123"
**When** 用户使用用户名 "alice" 和密码 "secret123" 调用登录接口
**Then** 返回状态码 200
 **And** 返回包含有效的 JWT token
 **And** 响应中包含用户基本信息（用户名、邮箱）

### SC02 - 密码错误
**Given** 用户名 "alice" 已注册
**When** 用户使用正确用户名但错误密码调用登录接口
**Then** 返回状态码 401
 **And** 响应中包含 "密码错误" 提示
 **And** 错误次数 + 1

### SC03 - 账户锁定
**Given** 用户名 "alice" 已连续输错密码 5 次，账户已锁定
**When** 用户使用正确密码调用登录接口
**Then** 返回状态码 423
 **And** 响应中包含 "账户已锁定，请 15 分钟后重试"
```

### 详细规范 4：测试方案

- 必须同时覆盖单元测试和集成测试
- 单元测试：列出需要测试的核心函数及其测试点（正常路径 + 异常路径）
- 集成测试：列出需要测试的完整流程和边界场景
- 必须说明如何运行测试（命令行、工具名称等）
- 必须明确测试的成功判定标准

#### 测试方案示例

```
## 测试方案

### 单元测试
- crypto.py:
  - test_hash_password: 验证 bcrypt 哈希生成、长度正确
  - test_verify_password: 正确密码返回 True，错误密码返回 False
  - test_verify_password_salt: 相同密码不同 salt 哈希不同

- auth.py (mock DB):
  - test_login_success: 正确用户名密码 → 返回 JWT
  - test_login_wrong_password: 密码错误 → 返回 401
  - test_login_locked: 账户锁定时登录 → 返回 423

### 集成测试
- 完整登录流程：连续 5 次输错密码 → 锁定 → 等 15 分钟 → 解锁后可登录
- 并发登录：同一账户同时多次登录 → 验证锁的线程安全

### 运行方式
```bash
pytest tests/ -v
pytest tests/integration/ -v --no-header
```

### 成功判定标准
- 所有单元测试通过
- 集成测试覆盖所有边界场景
- 覆盖率：核心模块 ≥ 90%
```

## Harness

编程任务敲定输出必须包含以下四个固定区块，结构固定，顺序不能调换：

```
## 1. 需求拆解

### 需求点
1. [需求点 1]
2. [需求点 2]
...

### 待确认问题
- [优先级] [问题描述]
...


## 2. 实现方案

[ ] 1. [任务描述]
    → 涉及：[文件/模块列表]
    → 依赖：[前置任务编号，无则写 "无"]
    → [高风险] [如有]

[ ] 2. [任务描述]
    ...


## 3. BDD 场景定义

### SCxx - [场景名称]
**Given** [前置条件]
**When** [触发动作]
**Then** [预期结果 1]
 **And** [预期结果 2]

### SCxx - [场景名称]
...


## 4. 测试方案

### 单元测试
- [模块名]:
  - [测试点 1]
  - [测试点 2]
  ...

### 集成测试
- [场景 1]
- [场景 2]
...

### 运行方式
```
[运行命令]
```

### 成功判定
- [判定标准 1]
- [判定标准 2]
...
```

## 附录：优先级说明

| 优先级 | 含义 |
| :--- | :--- |
| [高] | 不确认无法继续实现，必须先问用户 |
| [中] | 可以先做合理假设并在方案中标注，后续确认 |
| [低] | 不影响核心功能的细节，可后续迭代完善 |
```
