### 概要：Doc-Write 路由与架构概览

本 Skill 是文档协作系统的入口，负责两个职责：
1. **路由** — 根据用户意图，将请求分发到下方的内部 Skill
2. **概览** — 提供系统三层架构和两个内部 Skill 的职责描述

以下为本索引的逻辑结构：

1. 三层架构 — 系统层 / 规范层 / 业务层的关系
2. 路由规则 — 如何判断用户意图应交给哪个 Skill
3. 职责概览 — Architect 和 Framework 的分工描述
4. 交接说明 — 两个 Skill 之间的协作方式

### 详细规范 1：三层架构

文档协作系统由三个层次组成，底层规格与规范层正交，组合产出业务层文档：

```
L0 系统层  ← _Doc_System_Framework
│  定义：Role Profile 格式、Format 模板格式、渲染算法
│  产出：Roles/*.md、Formats/*.md
│
L1 规范层  ← _Doc_Architect_Skill
│  定义：写作方法论（交互协议、逻辑树、排版约束、术语注册）
│  约束：写作行为本身，不依赖具体角色或模板
│
L2 业务层  ← 最终产出的具体文档
│  输入：L0 的模板骨架 + L1 的写作方法论
│  产出：一份具体的业务文档
```

层间关系：

| 关系 | 说明 |
| :--- | :--- |
| **L0 → L2** | 模板提供骨架（structure map 定义有哪些 slot） |
| **L1 → L2** | 规范约束行为（写作时必须遵守交互协议、排版约束、术语注册） |
| **L0 → L1** | L1 引用 L0 的术语（Role Profile），但不被 L0 定义 |

关键区分：
- L0 回答"写什么结构"（底层规格）
- L1 回答"怎么写"（横切约束）
- 两者正交——不管程序员还是产品经理，写作方法论一致；不管技术方案还是会议纪要，模板结构不同

路由后的动作：
- `Read` 被路由到的内部 Skill 全文
- 按该 Skill 的指引执行操作
- 不再保留本索引文件的上下文

#### Doc_Architect_Skill — 写作规范（L1 规范层）

- **定位**：规范层——横切约束，约束所有写作行为
- **服务对象**：想写文档的人

- **产出物**：一份具体的文档

### 详细规范 4：交接说明

- **交接方式**：文件系统，不通过上下文传递
  - Framework 产出文件到 `Roles/` 和 `Formats/`
  - Architect 读取这些文件作为写作骨架
- **交接触发**：
  - Framework → Architect：模板产出后提示用户"可以基于此模板撰写文档"
  - Architect → Framework：发现模板缺失时建议用户"是否要先创建模板？"
- **跨界约束**：跨界操作必须征得用户同意，Agent 不得自动切换 Skill

> 完整协作协议见 `_Doc_System_Framework` 的"协作协议"章节。
