```yaml
Skill_Name: Doc-Writer
Purpose: 指导 Agent 协助用户撰写结构化技术文档
Execution_Trigger: 用户请求写文档、设计文档结构、整理技术资料时激活
```

## 概要

本 Skill 指导 Agent 协助用户撰写技术文档。

**书写三原则：**
- **依据翔实** — 对已有内容的充分把握，消除不确定
- **信息缩减** — 对读者需求的充分理解，不关心的不出现
- **最佳实践** — 以固定骨架与格式输出，风格统一可预期

**书写五规范：**
1. 意图锁定 — 确认文档类型、读者、边界
2. 文档骨架 — 用逻辑树预审文档结构
3. 固定格式 — 所有文档遵循统一骨架
4. 信息缩减 — 按读者需求裁剪内容
5. 单职拆分 — 超出单一职责时拆分为多文档

---

## 详细规范

### 详细规范 1：意图锁定

动笔前必须完成意图对齐，禁止直接输出正文。

**必须确认的信息：**
- 文档类型是什么（接口文档 / 设计文档 / 详解文档 / 埋点文档 / 测试文档 / 美术文档 / 需求文档 / 其他）
- **角色是谁**（CP-DEV / CPP-GAMESVR-DEV / 其他，对应角色目录名）
- 读者是谁（角色、知识背景）
- 文档边界：写什么、不写什么

**文档类型加载（带角色继承）：**
1. 确认文档类型 T 和当前角色 R
2. 从 `Skills/_Type-Definer.md` 的 YAML 类型定义块中读取类型 T → `base`
3. 检查 `Skills/_Type-Definer.{R}.md` 是否存在
4. 若存在，读取其中类型 T 的 YAML 定义，按 key 合并 → `merged = merge(base, override)`
   - override 中出现的 key 整体替换 base 同名 key
   - override 中未出现的 key 继承 base
   - override 可新增 key 扩展 base
5. 按 `merged` 的 `必须包含`、`格式约束`、`章节骨架` 等 key 约束写作
6. 若文档类型在基类和角色文件中均未定义，向用户询问关键约束并建议注册到 `_Type-Definer.md`

**镣铐起舞原则：**
- 确认角色 + 文档类型后，加载的 `merged` 约束即为"镣铐"
- 在严定的章节骨架、内容范围、格式约束下表达信息，做到信息充分缩减
- 禁止跳过角色确认步骤——不同角色的同名类型，骨架可能完全不同

**材料确认：**
- 动笔前必须充分阅读相关材料（如代码、设计稿、现有文档）
- 材料有缺失或存疑时必须立即向用户确认，禁止自行脑补
- 确认材料后方可进入写作

**执行方式：**
- 向用户提出最多 3 个针对性问题
- 呈现假设供用户确认
- 未对齐前禁止开始撰写

#### 用语 Harness

写作中禁止使用不确定语气词。出现以下词汇时 Agent 必须打回并要求补充材料：

| 禁止词汇 | 替代方式 |
| :--- | :--- |
| 或许、可能、大概 | 查证后给出确定结论，或标注"待确认：{具体问题}" |
| 不清楚、不确定 | 向用户提问，获取明确信息后再写 |
| 应该是、一般来说 | 给出确切条件或引用来源 |

需要强调的内容使用 **加粗** 标注。

#### 反模式

- 不要在未确定文档类型时就开始写——类型决定骨架
- 不要在未确定角色时就开始写——不同角色的同名类型骨架不同
- 不要在意图不明时就开始写"万能文档"

### 详细规范 2：文档骨架

每篇文档动笔前必须先输出逻辑树大纲，获得用户确认。

**逻辑树规则：**
- 用有序列表表达，每项不超过 15 字
- 项数控制在 3-7 项
- 逻辑树就是后续 `## 内容` 中各 `###` 标题的预览
- 用户确认后才能开始写正文

**冒号用法（最佳实践）：**
- 格式：`术语/总结：概述`，冒号前是核心概念，冒号后是对它的补充说明
- 用列表缩进进行分点详细解释
- 目的：读者扫一眼逻辑树就能抓住每个章节的核心概念

**预审循环：**
- 用户指出覆盖不全 → 重写逻辑树，重新确认
- 用户指出某项太重 → 建议拆分（见详细规范 5）

#### 示例

```
文档目标：API 接入指南
文档类型：程序接口文档

逻辑树：
1. 认证方式：API Key 与 OAuth 两种方案的选择
2. 请求格式：RESTful 规范与 payload 结构
3. 错误码：常见错误码含义与处理建议
4. 限流规则：速率限制与重试策略
```

### 详细规范 3：固定格式

所有文档必须遵循统一骨架，禁止自由发挥结构。

**文档骨架（Harness）：**

```yaml
# Harness: 文档骨架
header:
  Doc_Name: {文档名称}
  Purpose: {一句话说明，不超过 30 字}
  Type: {文档类型}              # 见 _Type-Definer.md
  Reader: {目标读者}            # 可选
  Version: {版本号}             # 可选
overview:
  summary: {1-2 句话说明本文档覆盖什么}
  tree:
    - {章节一标题}
    - {章节二标题}
    - {章节三标题}
content:
  - title: {与 tree 第 N 项一致}
    body: {正文}
appendix:                      # 可选
  glossary: {术语表}
  references: {参考资料}
```

**格式约束：**
- 禁止使用 H1（`#`）
- 标题层级最多 3 种：H2（`##`）/ H3（`###`）/ H4（`####`）
- 两个 `###` 标题之间必须隔一个空行
- 列表嵌套不超过 3 层
- 每个 `###` 章节不超过 500 tokens
- 文档总量控制在 1500-6000 tokens

### 详细规范 4：信息缩减

文档内容必须围绕读者需求裁剪，不关心的内容不出现。

**缩减规则：**
- 必须在动笔前确认文档类型 → 决定内容范围和格式风格
- 读者不需要知道的实现细节 → 删除或移入附录
- 读者已知的背景知识 → 不重复解释
- 每个章节只回答一个问题 → 多个问题拆分章节

**表述规则：**
- 结论前置，细节后置（信息首尾结构）
- 能用列表就不用散文
- 能用表格就不用长列表
- 术语首次出现时必须在附录术语表中注册

**复杂概念表述顺序（保护读者信息量）：**
- 第一步：简单概述 — 用 1-2 句话说明"这是什么"
- 第二步：术语解析 — 拆解关键术语，建立基础概念
- 第三步：详细解释 — 程序流程、数学推导、机制细节
- 第四步：外部引用 — 使用 markdown 引用语法（`>`）放置参考资料
- 数学公式必须使用 LaTeX（如 `$E = mc^2$`）

#### 反模式

- 不要写"为了完整性"而加入读者不需要的内容
- 不要在正文中解释读者已知的概念

### 详细规范 5：单职拆分

当一篇文档无法在单一职责内讲清楚时，必须拆分。

**拆分判断标准：**
- 逻辑树超过 7 项 → 拆分
- 单个 `###` 章节超过 500 tokens → 考虑拆分
- 文档试图回答 2 个以上独立问题 → 拆分

**拆分后的文档结构：**
- 主文档：索引 + 每个子文档的概述（1-2 句话）
- 子文档：各自独立，遵循单一职责

**加载规则：**
- 子文档以 `_` 前缀命名（如 `_Auth_API.md`）
- 子文档不被启动协议直接加载
- 需要时由 Agent 通过主文档索引按需读取

#### Harness：主文档索引格式

```yaml
# Harness: 主文档
header:
  Doc_Name: {主文档名称}
  Purpose: {整体目标}
  Type: {文档类型}
overview:
  summary: {整体覆盖范围}
  sub_docs:
    - name: {子文档 1 名称}
      file: _{子文档 1 文件名}.md
      summary: {一句话概述}
    - name: {子文档 2 名称}
      file: _{子文档 2 文件名}.md
      summary: {一句话概述}
```

### 术语表

| 术语 | 含义 |
| :--- | :--- |
| **逻辑树** | 文档的有序列表大纲，预览所有章节标题 |
| **信息缩减** | 按读者需求裁剪内容，不关心的不出现 |
| **单一职责** | 每篇文档只解决一个核心问题 |
| **主文档** | 索引多个子文档的入口文件，提供整体概述 |
| **子文档** | 被主文档索引的独立文档，`_` 前缀命名，按需加载 |
| **文档类型** | 文档的行文风格定义，存储于 `_Type-Definer.md`，写作时按类型加载对应约束 |
