## 依赖文档索引与角色声明
- 写作角色：文档架构师（Doc-Architect）
  - 写作偏好：结构极度严谨、逻辑链条清晰、严格受控于规范。
- 阅读角色：文档编写者 / AI Agent
  - 阅读偏好：步骤明确、规则定量化、易于模块化加载。

### 概要：文档架构规范逻辑树
本技能旨在指导 AI 如何严格按照单一职责和角色偏好来构建文档。以下为本规范的整体逻辑结构。阅读本规范的 Agent 可以根据当前的执行阶段，选择性地加载对应的章节部分。

1. 交互与意图锁定：通过固定化结构交互，确认读写角色与写作边界。
2. 逻辑树预审与拆分：先设计逻辑树，基于单一职责原则评估是否需要拆分文档。
3. 物理格式与排版约束：严格限制标题层级、列表深度，确保排版整洁。
4. 术语注册机制：发明术语必须在概要表中先注册，并受到严格的数量限制。
5. 信息首尾结构与引用：将易懂内容前置，难懂内容后置，参考资料使用引用块并放于尾部。

### 详细规范 1：交互与意图锁定协议
在正式动笔撰写任何内容之前，必须启动固定化的交互流程，严禁直接输出大篇幅正文。
- **角色引导与初始化：**
  - 必须引导用户从 `Doc-Writer` 中选择已有的角色作为“写作角色”和“阅读角色”。
  - 若所需角色的阅读/写作偏好尚未形成，须提示用户先更新 `Doc-Writer` 或提供外部个性文档索引。
- **固定化结构提问：**
  - Agent 须向用户提出明确的问题，要求其限制写作目的、核心重点。
  - 用户可在此阶段提供初始规则或粗略的逻辑树大纲。

### 详细规范 2：逻辑树预审与单职拆分标准
文档的骨架必须在动笔前获得用户确认，并遵循软件工程中的单一职责原则。
- **逻辑树预审流程：**
  - 根据用户的输入，Agent 必须先设计并输出“有序列表逻辑树大纲”。
  - 必须获得用户的明确同意后，才能开始撰写正文。
  - 若用户指出大纲内容覆盖不全、不均等问题，须立刻重写逻辑树并重新申请确认。
- **单职拆分红线：**
  - 每一个文档必须是“单一职责”的，只解决一个核心问题。
  - 若一个文档企图承担多个职责，Agent 必须主动向用户建议拆分为多个子文档。

### 详细规范 3：物理格式与排版约束
为了防止文档结构失控，必须在排版深度上设置硬性红线。
- **标题级别限制：**
  - 严禁使用一级标题（#）。
  - 同一文档内，标题等级最多只能使用 3 种（二级标题 `##`、三级标题 `###`、四级标题 `####`）。
- **列表层级限制：**
  - 有序与无序列表的嵌套不得超过 3 层。
  - 缩进次数最多为 2 次，超出此限制时必须考虑拆分文档。

### 详细规范 4：术语注册与准入机制
未经声明的术语会增加读者的认知摩擦，必须对其进行数量和流程的双重控制。
- **先注册后使用：**
  - 任何自创术语，必须在概要部分的“术语系统声明表”中注册后，方可在正文中被引用。
- **术语数量配额：**
  - **技术类文档：** 发明术语不得超过 15 个。
  - **非技术类文档：** 发明术语不得超过 6 个；发明术语与技术专业术语的总和不得超过 15 个。

### 详细规范 5：信息首尾结构与引用规范
合理分配标题内部的认知负载，确保不同阅读深度的读者都能快速获取信息。
- **内容组织顺序：**
  - **头部：** 放置需要阅读角色快速理解、开门见山的结论。
  - **中部与尾部：** 放置需要阅读角色深入理解的推导、细节、长文本。
- **引用块与附录使用：**
  - 凡是不需要阅读角色立刻理解的参考信息、外部索引，统一使用 Markdown 的 `>` 语法。
  - > 示例：这是一个指向外部依赖文件的引用展示。
  - 若依赖文档较多，允许在文档最后增加“附录”章节进行统一指向。
