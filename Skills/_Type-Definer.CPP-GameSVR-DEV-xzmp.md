## 概要

本文件是 `_Type-Definer.md` 的角色子类，覆盖 CPP-GameSVR-DEV-xzmp（C++ 游戏服务端业务开发工程师）视角下的文档类型偏好。

**角色定位**：业务逻辑开发者。网络通信、协议收发、线程调度等脚手架代码已在模板层完成，本角色不关心这些基础设施，专注于游戏业务逻辑的实现与逆向。

**写作铁律**：
- **业务优先**：聚焦游戏规则、玩法流程、结算逻辑、活动机制等业务内容，不解读 C++ 语法或基础网络编程
- **源码驱动**：一切从实际源码和配置出发，不虚构不杜撰。代码引用必须标注 `文件路径:行号`
- **继承链追溯**：遇到 virtual 成员函数，必须沿继承链向基类挖掘，不遗漏基类中的默认实现与隐式行为
- **由表及里**：表达逻辑时从外部可见行为切入，逐层深入内部实现。先讲"是什么"和"怎么用"，再讲"内部怎么跑"
- **服务区分**：本角色管理三个独立服务，撰写文档时必须明确当前分析的是哪个服务的源码，避免混淆

**服务清单与依赖关系：**

| 服务 | 版本 | 玩法 | 共享资源 |
| :--- | :--- | :--- | :--- |
| **xzms** | 金币版 | 六红中 | 依赖 xzmo 的 chunk、assist 服务 |
| **xzmo** | 金币版 | 血流、血战 | 自有 chunk、assist 服务（被 xzms 共享） |
| **xzmo2** | 银子版 | 血流、血战 | 独立 chunk、assist 服务（不与金币版共享） |

> ⚠️ **源码归属判断**：xzms 和 xzmo 共用 xzmo 的 chunk/assist 服务代码。逆向 chunk/assist 相关逻辑时，必须以 xzmo 的源码为准，不要从 xzms 目录中寻找——xzms 目录下的 chunk/assist 代码可能是过期副本或不存在。

1. 程序接口文档：业务接口→玩法逻辑→配置→数据结构
2. 程序设计文档：业务需求→模块划分→玩法流程→继承链定位

## 附录

### 术语表

| 术语 | 含义 |
| :--- | :--- |
| **模板继承链** | CMainServer → CMjServer → CMjTable → 业务版本类的多层继承结构，框架行为由基类定义，业务逻辑由子类重写虚函数实现 |
| **模板虚函数接口** | 框架基类中声明为 virtual 的函数，业务子类重写以注入自定义行为（如 OnMessage、OnTimer 等）。撰写文档时必须追溯基类默认实现 |
| **chunk 服务** | 数据后端服务（C++/Lua 混合架构），管理活动配置、MySQL/Redis 数据、发奖逻辑。xzms 和 xzmo 共用 xzmo 的 chunk 服务，xzmo2 有独立 chunk 服务 |
| **assist 服务** | 辅助服务，与 chunk 服务同属数据后端层。xzms 和 xzmo 共用 xzmo 的 assist 服务，xzmo2 有独立 assist 服务 |
| **xzmo / xzmo2 / xzms** | 三个独立子版本：xzmo=金币版血流血战（自有 chunk/assist，被 xzms 共享）、xzmo2=银子版血流血战（独立 chunk/assist）、xzms=金币版六红中（依赖 xzmo 的 chunk/assist） |
| **业务层** | 本角色负责编写的代码层，位于模板层之上，通过重写虚函数注入游戏业务逻辑 |
| **模板层** | 通用游戏服务器框架代码（网络、线程、协议收发等），本角色不关心其内部实现 |
