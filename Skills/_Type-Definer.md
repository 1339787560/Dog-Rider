```yaml
Doc_Name: Type-Definer
Purpose: 定义各文档类型的行文风格，供 Doc-Writer 按需加载
Reader: Agent（执行 Doc-Writer 时读取）
```

## 概要

本文档定义各文档类型的行文风格。写作者必须根据文档类型调整内容结构和表达方式。

1. 使用方式：Agent 如何读取和应用文档类型
2. 角色继承机制：基类与子类的合并规则
3. 类型定义：各文档类型的结构化行文风格

---

## 内容

### 使用方式

- 当 Doc-Writer 确认文档类型后，从本文档中查找对应的行文风格
- 若文档类型未在本文档中定义，向用户询问该类型的关键约束并建议注册
- 写作时必须遵循该类型的内容范围、术语深度、格式约束

### 角色继承机制

不同角色撰写同一文档类型时，关注点不同。本文档作为**基类**，角色特化文件作为**子类**，运行时合并加载。

**文件命名：** `_Type-Definer.{Role}.md`，Role 为角色目录名（如 `CP-DEV`、`CPP-GAMESVR-DEV`）。

**加载流程：**
1. 从本文档读取基类定义 → `base`
2. 检查 `Skills/_Type-Definer.{Role}.md` 是否存在
3. 若存在，读取其中同名类型的 `override` → `merged = merge(base, override)`
4. 按 `merged` 约束写作；若不存在，按 `base` 约束写作

**合并规则：**
- `override` 中出现的 key → 替换 `base` 同名 key（整体替换，不合并列表）
- `override` 中未出现的 key → 继承 `base`
- `override` 可新增 key → 扩展 `base`

**类型定义的固定 key（每个类型必须包含）：**

| key | 含义 |
| :--- | :--- |
| 定位 | 一句话说明目标读者和用途 |
| 必须包含 | 内容范围（必须出现的项） |
| 不包含 | 内容范围（明确排除的项） |
| 格式约束 | 排版和表达规则 |
| 术语深度 | 术语使用规则 |

**可选 key：** 章节骨架、复杂概念处理、示例、可选包含 等，按类型需要添加。

### 类型定义

以下为所有文档类型的结构化定义。每个类型是一个 YAML 对象，key 即为类型名称。

```yaml
程序接口文档:
  定位: 面向调用者，提供接口的完整使用说明
  必须包含:
    - 接口名
    - 用途（一句话说明）
    - 参数（名称、类型、必填、说明）
    - 返回结果（字段、类型、说明）
  不包含:
    - 实现原理
    - 内部调用链路
    - 设计决策背景
  格式约束:
    - 每个接口独立一个 `###`
    - 参数和返回结果用表格
    - 必须提供请求/响应示例
    - 错误码单独列表
  术语深度:
    - 使用行业通用术语，不解释基础概念
    - 项目特有字段名必须有说明列

程序设计文档:
  定位: 面向团队成员，接收策划需求，产出程序实现的技术选型和方案
  必须包含:
    - 需求摘要（策划需求的核心要点）
    - 技术选型（选了什么、为什么选）
    - 实现方案（模块划分、接口设计、数据结构）
    - 风险评估（边界条件、性能瓶颈、已知限制）
  不包含:
    - 策划需求的原文复制
    - 与实现无关的业务背景
  格式约束:
    - 技术选型用表格对比（方案 A / 方案 B / 结论）
    - 模块划分用 Mermaid 图或有序列表
    - 关键决策用 **加粗** 标注并附理由
    - 数学公式用 LaTeX
  术语深度:
    - 项目内部术语可直接使用
    - 跨模块术语需标注来源模块
    - 引入新技术时必须说明选型理由

程序详解文档:
  定位: 从已有程序及其配置出发，进行内容逆向，产出设计文档或流程解析
  必须包含:
    - 程序入口与整体结构
    - 核心流程（调用链路、数据流向）
    - 关键配置项的作用
    - 模块间的依赖关系
  不包含:
    - 逐行代码翻译
    - 无上下文的代码片段堆砌
  格式约束:
    - 先概述整体结构，再逐模块展开
    - 流程用 Mermaid 图或有序列表
    - 代码片段标注文件路径和行号范围
    - 配置项用表格（字段 / 默认值 / 作用）
  术语深度:
    - 逆向过程中发现的术语必须在附录注册
    - 引用源码时使用 `文件路径:行号` 格式
  复杂概念处理:
    - 第一步：用 1-2 句话说明模块职责
    - 第二步：用流程图展示调用关系
    - 第三步：展开关键代码段并注释
    - 第四步：引用相关设计文档或外部资料

数据埋点文档:
  定位: 面向数据分析师和开发团队，定义埋点方案及后续分析手段
  必须包含:
    - 埋点事件名
    - 触发时机（什么用户行为 / 什么系统状态触发）
    - 上报参数（字段名、类型、含义）
    - 分析 SQL（可直接执行的查询示例）
  不包含:
    - 埋点 SDK 的接入教程
    - 数据仓库架构说明
  格式约束:
    - 每个埋点事件独立一个 `###`
    - 触发时机用简洁的行为描述，不用代码伪逻辑
    - 上报参数用表格
    - SQL 用代码块，附注释说明查询目的
  术语深度:
    - 业务指标名必须与数据平台一致
    - SQL 中的表名和字段名必须有注释

测试文档:
  定位: 面向开发和 QA，提供可执行的测试用例集合
  必须包含:
    - 函数/接口单元测试：输入、预期输出、边界条件
    - 流程集成测试：端到端场景、依赖 Mock 策略
    - 客户端测试：UI 交互、状态变化、异常展示
  不包含:
    - 测试框架搭建教程
    - CI/CD 流水线配置
  格式约束:
    - 用 `情形` + `期望` 结构编写每个 case
    - 每个 `###` 对应一个测试模块
    - case 用有序列表，情形和期望用缩进嵌套
  术语深度:
    - 测试术语（Mock、Stub、断言）可直接使用
    - 业务术语需与需求文档一致
  示例: |
    ### 登录接口测试

    1. 情形：正确用户名和密码
       期望：返回 200，body 包含 token

    2. 情形：密码错误
       期望：返回 401，body 包含 error_code: "INVALID_CREDENTIALS"

    3. 情形：用户名为空
       期望：返回 400，body 包含 error_code: "USERNAME_REQUIRED"

美术文档:
  定位: 面向美术和前端，提供视觉实现的精确规格
  必须包含:
    - 整体示意图（截图或设计稿链接）
    - 各元素的位置（坐标 / 布局规则）
    - 字号、字重、颜色值
    - 间距和尺寸规格
  可选包含:
    - 设计理念和审美讨论
    - 用户研究数据
  不包含:
    - 前端实现代码
    - 交互逻辑的代码细节
  格式约束:
    - 颜色值用 HEX 或 RGBA，附色块示意
    - 字号和间距用表格
    - 位置描述用结构化标注（如 `距左: 16px, 距顶: 24px`）
    - 必须附示意图或设计稿引用
  术语深度:
    - 设计术语（间距、字重、圆角）可直接使用
    - 组件名需与前端代码中的命名一致

产品需求文档:
  定位: 面向全团队，讲清楚做什么、为什么做、怎么验证效果
  必须包含:
    - "为什么做"（业务背景、用户痛点、预期收益）
    - "做什么"（功能范围、交互流程、验收标准）
    - "怎么验证"（上线后观察指标、数据埋点方案、调优策略）
  不包含:
    - 技术实现方案
    - 代码层面的细节
  格式约束:
    - "为什么做"用数据或用户场景支撑，不用主观判断
    - 功能范围用列表，交互流程用有序步骤或流程图
    - 验证指标用表格（指标名 / 当前值 / 目标值 / 数据来源）
    - 调优策略需明确触发条件和调整动作
  术语深度:
    - 业务术语必须在首次出现时解释
    - 禁止使用技术术语，如必须使用需附通俗解释
```

## 附录

### 类型注册规范

新增文档类型时，按以下结构添加到"类型定义"的 YAML 块中：

```yaml
{类型名称}:
  定位: {一句话说明该类型的目标读者和用途}
  必须包含:
    - {必须包含的内容项}
  不包含:
    - {明确排除的内容项}
  格式约束:
    - {排版和表达规则}
  术语深度:
    - {术语使用规则}
```

可选 key（按需添加）：`章节骨架`、`复杂概念处理`、`可选包含`、`示例`。
