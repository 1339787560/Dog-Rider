## 概要

本文件定义跨角色、跨服务协作场景下的文档类型。当一个功能涉及多个服务（如 Creator 客户端 + CP 礼包服务 + CPP 游戏服务）时，使用本文件中的类型定义来组织文档。

**使用场景**：
- 好友房功能：Creator 客户端 ↔ CP 礼包服务 ↔ CPP 游戏服务
- 充值流程：Creator 客户端 → CP 礼包服务 → CPP 游戏服务
- 活动系统：CP 礼包服务 ↔ CPP 游戏服务 ↔ Creator 客户端展示
- 任何需要跨多个角色代码才能完整理解的功能

**写作铁律**：
- **先总后分**：先讲清楚多方协作的完整流程（谁发给谁、什么顺序、什么数据），再对每一方各自的工作做详解
- **消息驱动**：跨服务协作的本质是消息传递。文档必须以消息流为主线，不遗漏任何一跳
- **边界清晰**：每个服务的职责边界必须明确标注——哪些是本服务负责的，哪些是依赖其他服务的
- **源码驱动**：每一方的逻辑必须从实际源码逆向，标注 `文件路径:行号`，不虚构不杜撰

1. 跨服务程序详解文档：协作总览→消息流→各方详解→数据结构→术语
2. 跨服务接口文档：消息协议总览→各方接口→数据结构
3. 跨服务设计文档：需求→协作流程→职责划分→接口协议→风险

## 内容

### 类型定义

```yaml
跨服务程序详解文档:
  定位: 从多方源码出发，逆向产出跨服务协作的完整解析，供各角色开发者理解整体流程和各自职责
  必须包含:
    - 协作总览（涉及哪些服务、各自角色一句话、整体协作流程概述）
    - 消息流全链路（从用户操作触发到最终结果返回的完整消息流，跨服务逐跳标注，每跳标注 `文件:行号`）
    - 各方详解（按服务分章节，每个服务独立详解其内部处理逻辑，沿继承链/调用链展开）
    - 数据结构对照（同一数据在不同服务中的表示形式差异，如 Protobuf ↔ JSON ↔ TypeScript interface 的转换）
    - 术语表与坑点（跨服务术语定义、协议歧义、版本兼容性陷阱）
  不包含:
    - 单一服务内部的纯技术细节（如 C++ 模板继承链的完整追溯、CocosCreator 渲染管线细节）
    - 与跨服务协作无关的模块逻辑
    - 基础知识讲解
  格式约束:
    - 文档开头必须列出涉及的服务清单（服务名 / 角色 / 源码路径）
    - 协作总览用 Mermaid sequenceDiagram 或时序图，标注每条消息的服务间方向
    - 消息流全链路用有序列表，每个步骤标注 `服务名 — 文件:行号 — 函数名 → 业务含义`
    - 各方详解按服务分 `##` 章节，每个服务内部按"接收→处理→发送"顺序组织
    - 数据结构对照用并列表格（字段 / CP 侧 / CPP 侧 / Creator 侧）
    - 代码引用必须以 markdown 引用形式标注出处（`文件路径:行号`）
    - 坑点用"现象 → 原因 → 解决"三段式，附涉及的服务和代码位置
  术语深度:
    - 跨服务协议名可直接使用，首次出现时附消息方向和服务归属
    - 各服务内部术语首次出现时附一句话说明
    - 同一概念在不同服务中有不同命名时，必须在术语表中建立映射
  复杂概念处理:
    - 第一步：用 1-2 句话说明该协作环节的整体职责
    - 第二步：用时序图展示多方消息交互
    - 第三步：逐服务展开内部处理逻辑
    - 第四步：标注关键代码引用出处
  章节骨架:
    - 协作总览（服务清单 + 整体流程概述 + 时序图）
    - 消息流全链路（跨服务逐跳消息流，从用户操作到最终结果）
    - 各方详解（按服务分章节：Creator 客户端 / CP 礼包服务 / CPP 游戏服务）
    - 数据结构对照（同一数据在不同服务中的表示差异）
    - 术语表与坑点（跨服务术语映射 + 协议歧义 + 版本兼容性陷阱）

跨服务接口文档:
  定位: 面向全团队，提供跨服务协作的消息协议和接口定义
  必须包含:
    - 协作总览（涉及哪些服务、消息流向概述）
    - 消息协议总览（所有跨服务消息的汇总表：消息名、方向、数据结构、触发时机）
    - 各方接口（按服务分章节，每个服务的对外接口定义）
    - 数据结构（跨服务传递的数据结构定义，标注各服务侧的表示形式）
    - 错误码与异常处理（跨服务调用的错误码定义和异常处理策略）
  不包含:
    - 各服务内部实现细节
    - 单服务内部的事件/接口
    - 基础知识讲解
  格式约束:
    - 文档开头必须列出涉及的服务清单
    - 消息协议用表格（消息名 / 方向 / 数据结构 / 触发时机 / 处理方）
    - 各方接口按服务分 `##` 章节，接口用 TypeScript/Protobuf/C++ 代码块 + 字段说明表格
    - 数据结构用并列表格（字段 / CP 侧类型 / CPP 侧类型 / Creator 侧类型）
    - 错误码用表格（错误码值 / 名称 / 含义 / 处理建议）
  术语深度:
    - 跨服务协议名可直接使用
    - 各服务内部术语首次出现时附一句话说明
  章节骨架:
    - 协作总览（服务清单 + 消息流向概述）
    - 消息协议总览（跨服务消息汇总表）
    - 各方接口（按服务分章节）
    - 数据结构（跨服务数据定义）
    - 错误码与异常处理

跨服务设计文档:
  定位: 面向全团队，产出跨服务协作功能的设计方案
  必须包含:
    - 需求摘要（功能需求要点，用业务语言描述）
    - 协作流程设计（多方消息交互的时序图 + 有序列表）
    - 职责划分（每个服务负责什么、不负责什么，用表格对比）
    - 接口协议设计（跨服务消息的定义：消息名、方向、数据结构）
    - 数据流设计（数据在各服务间的流转和转换方式）
    - 各方实现要点（每个服务需要实现的关键逻辑概述）
    - 风险评估（跨服务调用的超时、重试、降级策略）
  不包含:
    - 单一服务内部的详细设计
    - 策划需求原文
    - 基础设施设计
  格式约束:
    - 文档开头必须列出涉及的服务清单
    - 协作流程用 Mermaid sequenceDiagram 或时序图
    - 职责划分用表格（职责项 / CP 服务 / CPP 服务 / Creator 客户端）
    - 接口协议用表格（消息名 / 方向 / 数据体 / 说明）
    - 数据流用 Mermaid flowchart 或有序列表
    - 关键决策用 **加粗** 标注并附理由
  术语深度:
    - 跨服务协议名可直接使用
    - 引入新的跨服务概念时必须用一句话定义
  章节骨架:
    - 需求摘要
    - 协作流程设计（时序图 + 消息流）
    - 职责划分（各服务职责对比表）
    - 接口协议设计（跨服务消息定义）
    - 数据流设计（数据在各服务间的流转）
    - 各方实现要点（各服务关键逻辑概述）
    - 风险评估（超时、重试、降级策略）
```

### 加载规则

本文件作为**基类**，不绑定特定角色。加载时机：

1. 用户任务涉及多个服务的协作逻辑时（如"好友房功能详解"、"充值流程文档"）
2. Doc-Writer 确认文档类型为"跨服务程序详解文档"、"跨服务接口文档"或"跨服务设计文档"时
3. 先从本文件读取类型定义 → 再检查各角色的 `_Type-Definer.{Role}.md` 是否有 override → 合并后执行

### 与角色 _Type-Definer 的关系

| 文件 | 职责 |
| :--- | :--- |
| `_Type-Definer.md` | 全局基类，定义通用文档类型 |
| `_Type-Definer.{Role}.md` | 角色特化，覆盖单角色视角下的文档类型偏好 |
| **`_Type-Definer_Inter.md`** | 跨服务特化，定义多服务协作场景的文档类型 |

### 术语表

| 术语 | 含义 |
| :--- | :--- |
| **跨服务协作** | 一个功能的完整实现涉及多个服务（Creator 客户端 / CP 礼包服务 / CPP 游戏服务），需要结合多方代码才能完整理解 |
| **消息流** | 跨服务协作中的数据传递路径，以消息（协议包/HTTP 请求/事件）为单位在服务间流转 |
| **职责边界** | 每个服务在协作中负责的范围，超出边界的逻辑由其他服务处理 |
